Introduction

This document describes the coding style guidelines and procedures to be used by the IE Development Team.

By standardizing on a set of coding styles, method and function implementations in the project will tend toward the same structure. This commonality will help reduce coding errors and speed up code review times since the team will be able to visually identify coding patterns that pertain to the majority of error handling scenarios.

Both the error handling and basic code layout guidelines derive from the experience of previous product development groups. While no guideline will satisfy everyone, it is important to remember the goal of commonality for readability, performance and robustness.

Philosophy

Coding standards exist to ensure that a large code base, maintained by dozens or hundreds of developers, remains uniformly easy to read and maintain. Coding standards do not exist to force people to comply with rigid, arbitrary rules.

 

The following precepts should guide your use of this document:

 

  • Follow the standard.
  • Use common sense. If a rule doesnt make sense for your situation.  Discuss it with mailto:IECoding.
  • Own the standard. If you cant follow it, in most cases you should:
    • Draft a refinement to the standard.
    • Solicit opinions (unless clearly not controversial or of concern to others). (Instructions go here.)
    • If theres agreement, update the standard. (See section process for changing/add the standards)
    • Otherwise, update the standard to reflect the different viewpoints.
      • We might be able to pick one later, so documenting it would help.
      • Even if we dont agree on one right way, we might still agree on many wrong ways.

 

Process for Changing/Adding to the Standards

Since there is so much existing code, we will adopt the following approach to introducing the new standards:

 

  • Existing code need not change.
  • You MAY change existing code opportunistically if you wish.
  • When opportunistically reformatting existing code to conform to new guidelines, do it at least on a per-function basis, and preferably on a per-file basis. Do not mix styles within a single function.

  • All new code should conform to the standards and guidelines in this document.

 

On some matters, one size does not fit all, and each project must make its own decisions.

 

There are many points where this document makes a recommendation, but explicitly leaves it up to each project to decide.

If you have standard qualifiers and macros, make your suggestion to mailto:IECoding.

Compliance to Standards

All new code must conform to these standards.

If you substantially modify existing code, it must also conform to these standards.

It is recommended that non-substantial modification also conform to these standards.

 

These standards apply to all of IEs components expect private copies of external code (such as the image filters). For modifications to copies of external code, use the ambient style of the existing code.

File names

  • C files and their headers are to end with the extensions c and h, respectively.
  • C++ files and their headers are to end with the extensions cpp and hpp, respectively outside of inetcore and cxx and hxx, respectively within inetcore.
  • C++ files should be named after the class (main class) they implement (without the leading C in the name).
  • Like variables names, file names should not be acronyms or abbreviated. We no longer live in an 8.3 world.

 

New files must comply with standards. Existing files need only be migrated when being restructured.

General Code Layouts

The following rules list some guidelines on general coding layout. They are intended to make everyone's code look as similar.

The most basic guideline is:

         Vertical and horizontal spacing is good

         Crunched up code is bad.

         Use tabs (expanded to spaces, see below) and the return key liberally to space out your code into logical, readable segments

         Keep common elements together

Tabs and Column Width

Set indentation to every 4 characters for all types of source files. You must also set your editor to expand tabs to spaces.

You should restrict each line of code in your source files to 80 columns.

Comments and documentation

Comments are essential to the readability of code. We embed most of the detailed documentation about how a given module of code works in the comments contained in it, rather than in an external document. The goal of this is to make sure that the documentation stays synchronized with the code it attempts to document. When code is changed, the comment accompanying it should be checked to make sure it still applies.

There are two important things to remember:

Grammar: Write comments in properly formed English sentences. Never use profanity or make negative references to other people, companies or products.

 

 

Meaningfulness: Write comments that describe the higher level semantics of the code. Do not just repeat the function of the code in the comments:

 

 

// Increment counter.  <- NO

++count;

We use two kinds of comments, banner comments and inline comments. Banner comments are used to document top-level objects in the code: a source file, functions, class definitions, structure definitions, typedefs and global variables. As a rule of thumb, all but the most trivial functions should have a banner comment. So should all classes, all structures, and all global variables except those used purely internally in a module.

The following examples show the minimal requirements. Additional entries, such as History or Note, are always acceptable.

Banner Comments

A banner comment for a source file should look like this:

//+---------------------------------------------------------------------------

//

//  Microsoft Internet Explorer

//  Copyright (C) Microsoft Corporation, All rights reserved.

//

//  File:       Timer.cxx

//

//  Contents:   Class implementation for CTimerMan, CTimer,

//              and CTimerSink, which support non-Windows based timers

//              for better service and granularity.

//              Timer runs on separate thread, posting messages to main

//              window proc. Message is not WM_TIMER, however, so that

//              there won't be any delay in posting the message.

//              See Timers and Synchronization spec for details.

//

//  Created:    Dec. 5, 1996

//

//--------------------------------------------------------------------------

 

Description is sometimes called Abstract.  History is optional, but recommended if you can handle it.  History is not intended to chronicle each small change to the file, but should contain entries of significant changes to the file.  An example of a significant change would be change threading support from single to multi or vice versa.  That entry should be similar if not the same as the relevant part of the change description when checked-in.

 

Notes:

  • In the History section, only mention major changes. Dont make a History entry for every bug fix.

  • In the History section, put the most recent entry at the top, so that the entries occur in reverse chronological order.

  • In the History section, the first entry should be Created. In cases where you have simply copied the file from another project (with only minor modifications), you might say Derived from xxx.

 

Rationale:

  • The + in the first line of the file banner comment is for consistency with function banner comments.

  • There is no Module Name or File Name section in the header. All editors that we know of show the file name in the title stripe of the document window or the application window.

  • There are two spaces between the // and each section title, so that you can simply tab from the trailing / to the start of the section title.

  • The copyright notice is in the format specified by LCA.

  • The choice of the word Abstract (as opposed to Description) is meant to emphasize that we want a short description here so you can tell at a glance whats in the file. Put details in the Notes section.

  • The format of the History section looks odd at first, but we think its actually easier to use than a format where the descriptive comment starts on the same line as the date. In the format shown here, the indentation to the descriptive comment is always the same, and its obvious how far to indent when the description wraps.

  • Since the first entry in the History section is Created, there is no need for an Author section.

 

A banner comment for a function or method should look like this:

//+----------------------------------------------------------------------------

//

//  Method:     CreateTimer         [ITimerService]

//

//  Synopsis:   Creates a CTimer object

//

//  Arguments:  [in] pReferenceTimer

//

//                   Timer to base new timer off of. This is so

//                   the behavior of the new timer is effected

//                   by the reference timer. E.g., if reference

//                   timer is frozen, so will the new timer.

//                   Can be NULL, to use default timer.

//

//              [out] ppNewTimer - Returned timer.

//

//  Returns:    S_OK

//              E_OUTOFMEMORY: either ran out of memory or hit the max allowed

//              E_POINTER

//              E_INVALIDARG

//

//-----------------------------------------------------------------------------

This does not include argument comments.  Argument comments are included in the function header as shown below. Also note the use of IN and/or OUT for pointer/reference arguments.

HRESULT CForm::InsertNewAt(

    CLSID        clsid,   // Class of the embedding to insert

    IN RECT*     prc,     // Pointer to logical rectangle where embedding

                          // is to be located

    DWORD        dw,      // Aspect of the embedding, usually DVASPECT_CONTENT  

                          // but also could be DVASPECT_ICON

    OUT CSite**  ppSite)  // Where the site of the new embedding is returned

For over-ridden methods, the banner comment should indicate what class or interface introduced the method, by following the member name with that class or interface.

Any method or function which can fail with side-effects should have those side-effects clearly communicated in the Notes portion of the banner comment. As a general rule, code should be written so that it has no side-effects in error or failure cases; the presence of such side-effects should have some clear justification when the code is written.

(Such justification is not necessary for routines which zero-out or otherwise overwrite some output-only parameter.)

Inline Comments

In addition, sometimes it is useful to write more detailed comments inside a function or method to help describe what is going on inside it. The alternative // commenting style should be used for this type of comment.

{

    //

    //  The following section of code is tricky (3)

    //

 

    if (3 == n1)

    {

        n2 = 4;         //  Whoa!  Four!  (1)

    }

    else

    {

        n3 = 5;         //  Whoops!  Five!  (1)

    }

 

    //  Calculate the factorial of n1 (2)

    n1 = NFactorial(n1);

}

 

 

End line comments (Style 1): End line comments are OK when they are short and meaningful.

 

Rationale:

  • There is limited room on the end of a line to write a meaningful comment.

  • Indentation of end line comments is inconsistent and gives code a ragged appearance.

Block code comments (Style 3): The preferred style for comments is the paragraph style: Comments describing a block of code should appear on a line by themselves, indented as the code they describe, with blank lines both before and after.

 

Notes:

  • Note the leading and trailing empty // lines.

  • Indent paragraph style comments and single line comments to the same level as the code they apply to.

  • The length of the text in a comment line should not exceed 60-80 columns

 

Rationale:

  • Liberal use of white space improves code readability.

  • Excessive line length harms readability.

 

C style comments: Do not use C style comments.

 

Rationale:

         The // at the beginning of each C++ style comment line clearly identifies each comment block.

 

Exceptions:

 

  • Use a C style comment to comment out the name of an unused method parameter:

        // Options that affect rendering.

But be cautious about ambiguity, as explained in the Note about ambiguous parameter comments.

TODO/HACK/BUG Comments

There needs to be a bug or work item for each TODO to the code base, which cannot without the outstanding issue being addressed.

 

The term hack should not be used. Besides, hacks needing to be avoided, it does not reflect well on Microsoft, when the source is made public.

 

BUG comments require a bug number and database name.

Please do use the predefined macros for argument a return type checks.

CHECK_ALLOC

Set hr to E_OUTOFMEMORY and jumps to Cleanup if NULL

CHECK_HR

Set hr and jumps to Cleanup if it hr FAILED

CHECK_PTR

Set hr to E_POINTER and jumps to Cleanup if NULL

CHECK_PTR_ARG

Set hr to E_INVALIDARG and jumps to Cleanup if NULL

THR

Assigns hr and traces the value

RRETURN

Ensures permitted success values are returned.

RRETURNX

Same as RRETURN except defines a new HRESULT list

When its OK to not use THR

As a general rule, it is always best to use THR.  Omitting it can make it much more difficult to determine the origin of failed HRESULTs.  However, there are times when skipping the macros makes sense.  Specifically, when checking a HRESULT that can fail during normal execution and the failure is handled by the function it occurred in (i.e., not returned from it).  This prevents the developer from having to define HRESULT lists for HRs that are never returned from the originating function. 

 

Below are a few common examples of these situations:

 

  • While loops and/or searching through lists

 

do

{

    hr = GetNextItem(..);   // Returns E_FAIL when the end is reached

while (SUCCEEDED(hr));   

 

  • Failure is handled within the function

o       Handled Failure

if (SUCCEEDED(hr))

{

    // Do Something

}

else

{

    // Handle failure

    ...

    // Failure was handled

    hr = S_OK;

}

o       Handled QueryInterface Call

if (SUCCEEDED(pObj->QueryInterface(IID_IMILBitmap, &pBmp)))

{

    // Do Something

}

else

{

    // Handle failure

    ...

    // Failure was handled

    hr = S_OK;

} 

 

 

For all except the simplest cases, THR should be used on all HRESULTs that are returned out of the function they originated in.  This is due to the fact that HRESULTs tend to be blindly propagated up the stack via IFC or similar constructs, and it is much more difficult to track down the origin of a FAILED HRESULT once it has been returned from the method it originated in.

 

If a failure HRESULT is returned during normal execution and shouldnt cause the instrumentation to trigger a failure, HRESULT lists can be defined to configure the macros to ignore them.  Doing this also forces us to document the return paths of error codes we are familiar with where documentation is most useful the code.  It allows us to break on other HRESULTs that are not explicitly specified which may be inadvertently introduced in future checkins.

1.      Subtypes,

3.      Most other general methods including virtual methods,

4.      Member variables

5.      Debug only subtypes, methods, and members, respectively.

Method and Function Size

Methods and functions, like a paragraph in an essay, should have one idea/task. If task has multiple parts it should be multiple methods the equivalent of an essay, not a paragraph.

 

There is no set size, because a large method might consist of a single switch statement and thus be logically simple.

 

Local Variable Declarations

Local variables should be initialized:

 

  • Return codes need to be set to failure by default.
  • Pointers should be set to NULL by default

Local variables should be declared exclusively at the top of a scope block, and preferably at the top of the function/method block. Inline declarations of variables should not be used, primarily for two reasons. The first is that inline declarations are difficult to read, particularly in busy code. It can be difficult to have to search through an entire method to find the type or initialization of a local variable. By placing at the top of the function, everyone knows exactly where to find these. If you must declare a variable within a particular scope, again declare it immediately after the opening brace.

The second reason has to do with our style of error handling which will be discussed later. Since we will be using goto statements in inetcore code base to isolate our error handling and cleanup code, following these rules of local variable declarations will help avoid compiler warnings of skipped variable initializations.

Commenting out code

Actually, we don't use comments to do this. Instead, surround the no-longer-used section of code with an #ifdef (see below). The #ifdef NEVER should start in column 1, as should the corresponding #endif - again, for readability and ease in locating these items.

Naming Conventions

Variables

The most common type of identifier is a variable name. All variable names are composed of three elements: prefixes, base type, and qualifier (base types are also referred to as tags). Not all elements are present in all variable names; the only part that is always present is the base type. This type should not be confused with the types supported directly by the programming language; most types are application specific. For example, an lbl type could refer to a structure containing symbol information; a co could be a value specifying a color.

Base Types (tags)

As the above examples indicate, tags should be short (typically two or three letters) and somewhat mnemonic. Because of the brevity, the mnemonic value will be useful only as a reminder to someone who knows the application, and has been told what the basic types are; the name will not be sufficient to inform (by itself) a casual viewer what is being referred to. For example, a co could just as easily refer to a geometric coordinate, or to a commanding officer.

As is probably obvious from the above, it is essential that all tags used in a given application be clearly documented. This is extremely useful in helping a new programmer learn the code; it not only enables him (or her) to decode the otherwise cryptic names, but it also serves to describe the underlying concepts of the program, since the data types tend to determine how the program works. It is also worth pointing out that this is not nearly as onerous as it sounds; while there may be tens of thousands of variables in a program, the number of types is likely to be quite small.

Prefixes

Base types are not by themselves sufficient to fully describe the type of a variable, since variables often refer to more complex items. The more complex items are always derived from some combination of simple items, with a few operations. For example, there may be a pointer to an lbl, or an array of them, or a count of co's. These operations are represented in Hungarian by prefixes; the combination of the prefixes and the base type represent the complete type of an entity. Note that a type may consist of multiple prefixes in addition to the base type (e.g. a pointer to a count of co's); the prefixes are read right to left, with each prefix applying to the remainder of the type.

In theory, new prefixes can be created, just as new types are routinely created for each application. In practice, very few new prefixes have been created over the years, as the set that already exists is rather comprehensive for operations likely to be applied to types. Prefixes that have been added tend to deal with the specifics of machine architecture, and are variations on existing prefixes (i.e. different flavors of pointers). One can go overboard in refusing to create a new prefix, however; some new concepts really are logically expressed as prefixes, not types.

Qualifiers

While the prefixes and base type are sufficient to fully specify the type of a variable, this may not be sufficient to distinguish the variable. If there are two variables of the same type within the same context, further specification is required to disambiguate. This is done with qualifiers. A qualifier is a short descriptive word that indicates for what the variable is used. In some cases, multiple words may be used. Some distinctive punctuation should be used to separate the qualifier from the type; in C and other languages that support it, this is done by making the first letter of the qualifier upper-case. (If multiple words are used, the first letter of each should be upper-case; the remainder of the name, both type and qualifier, is always lower-case.)

Qualifiers are not to be abbreviated unless they documented as standard qualifier in this document.

Similarly qualifiers are not to be reduced to acronyms, unless it is documented as a standard abbreviation in this document.

Exactly what constitutes a naming context is language specific; within C the contexts are individual blocks (compound statements), procedures, data structures (for naming fields), or the entire program (globals). As a matter of good programming style, it is not recommended that hiding of names be used; this means that any context should be considered to include all of its subcontexts. (In other words, don't give a local the same name as a global.) If there is no conflict within a given context (only one variable of a given type), it is not necessary to use a qualifier; the type alone serves to identify the variable. In small contexts (data structures or small procedures), a qualifier should not be used except in case of conflict; in larger contexts it is often a good idea to use a qualifier even when not necessary, since later modification of the code may make it necessary. In cases of ambiguity, one of the variables may be left with no qualifier; this should only be done if it is clearly more important than the other variables of the same type (no qualifier implies primary usage).

Since many uses of variables fall into the same basic categories, there are a few standard qualifiers. If applicable, one of these should be used, since they specify meaning with no chance of confusion. In the case of multiple word qualifiers, the order of the words is not crucial, and should be chosen for clarity; if one of the words is a standard qualifier, it should probably come last (unfortunately, this suggestion is by no means uniformly followed). Functions

Unfortunately, the simple rules used for variable names do not work as well for Functions. Whereas the type of a variable is always quite important, specifying how that variable may be used, the important part of a Function is typically what it does. In addition, the context for Functions is usually the entire program, so there is more chance for conflict. To handle these issues, a few modifications are made to the simple rules:

1.      All Function names are distinguished from variable names by the use of some standard punctuation; in C this is done by capitalizing the first letter of the Function name (all variable names begin with a lower case type).

2.      In IE Hungarian, the return type of the function does not prefix the Function name.

3.      If the Function is a true function (operating mainly on its parameters and returning a value, with few or no side effects), it is standard to name the Function AFromBCD..., where A is the value returned, and B, C, D, etc. are the objects referred to by the parameters. For example,

CForm::LogicalFromPhysical

Sometimes true functions are also named BCDToA, but this practice is discouraged.

4.      If the Function is not a true function, use a few words to describe what the Function does (a verb followed by an object is usually good). Each word should be capitalized.

Macros

Macro names should be constructed using the same rules as Functions but be entirely in uppercase.

#define RRETURN(x) return x

Structures

Structure names should be entirely in uppercase, and their tags in lowercase. In C++, a structure name does not need to be typedef'd. The members of a struct should be named exactly as variables. For example:

Classes

Class names should be prefixed with the letter 'C', and be in mixed case. The tag of a class should be in mixed case, and omit the initial letter 'C'. Typically, the tag for a class includes the first letter of each word in the class, with sufficient disambiguating lower case letters.

Member variables of a class should be named as ordinary variables, but prefixed with an "_".

Static member variables should be prefixed with "s_".

Member functions should be named like ordinary functions, with no special prefix.

Classes and COM

Classes that implement COM objects should have their destructor private (or protected if you expect people to derive from you). This avoids multiple ways to delete objects.

Classes that implement COM object should initialize the _cRef to 1 on construction. (note, ATL does not do that). This makes it impossible to  be in the state where the class exists but can not be released.

Classes that implement IUnknown should place the Destructor in the Protected section of the class declaration so that only that class is able to destroy the object through it's final Release call.

Simple types (constants, enumerations, macros, structures, and typedefs) have type names all in uppercase, and tags all in lowercase. When a type name is entirely in uppercase, the words within the name may be separated by an underscore ("_").

Rationale for the convention on enumerated types and constants:

 

Qualifying enum constant names with the enum type name makes them less likely to collide: ColorRed is better than just Red. But when both the type name and the constant name consist of multiple words, its hard to unambiguously separate the enum type name prefix (Color) from the constant name (Red). Making the enum type name into a namespace name retains the benefit of the qualified name while unambiguously distinguishing the "type" part of the name from the "constant" part of the name: Color::Red.

 

Rationale for the convention on underscore-separation of words in ALL_CAPS:

 

Omitting the underscore can lead to ambiguity. Does NODEFAULT mean no default or node fault? Does SPI_GETWINDOWSEXTENSION mean get windows extension or an unfortunate double entendre?

 

Notes:

  • The use of #define is discouraged. They have no defined scope, and they cant be examined in the debugger. Use enums or const variables for constants, inline functions instead of macros.

  • The use of ALL_CAPS is restricted to preprocessor symbols. Rationale: The preprocessor isnt namespace aware, so there needs to be a standard to separate dangerous symbols (#defines) from well behaved ones (everything else).

 

  • Use caution with enums: The size and signedness of enum types is compiler implementation-defined. Therefore, if you are defining a set of constants that include large values (values that take more than 8 bits to represent), or a set of flags, consider defining them as const values of an explicitly sized type such as __int32. When defining a set of small constants, prefer enums, because they are generally more readable, and because erroneous assignments between an enum and an integral type can be detected by the compiler or a tool such as Lint.

Version numbers and code names are not to be part of any name. Version numbers and code names make code hard to port, reuse, and maintain.

Parameter declarations

Do not pass OUT parameters by reference because it makes it hard to determine whether a variable is modified across a call.

For example, consider

Function()

{

   int n = 3;

   Mystery(n);

   ASSERT(n == 3);

}

Is the assertion valid? If you missed the declaration of the function Mystery, you might think the answer is, "Of course it is. The value of n is never modified."

However, if the function Mystery were declared as

void Mystery(int &n);

then the answer is, "Maybe, maybe not."

If the Mystery function intends to modify its argument, it should be rewritten as

void Mystery(int *pn);

 

Function()

{

   int n = 3;

   Mystery(&n);

}

It is now clearer that the Mystery function can change its argument.

If you choose to pass an object by reference (for example, because it is a structure), either pass it explicitly by pointer (if it is an OUT parameter), or use a const reference (if it is an IN parameter). The const attribute indicates that the object is not modified by the function.

This preserves the rule that "a parameter passed without a & is not modified by the function." We have defined macros for common cases of object types, like REFCLSID, REFIID and REFPROPERTYKEY.

Standardizing Error Handling

Inetcore standard

By establishing guidelines for structuring our code around error conditions, it should become easier to review and maintain code written by others. Having these guidelines also makes writing code faster since, with some practice, error handling scenarios can be quickly categorized into a known, familiar set.

Scenario 1: Simple Error Return Values

The simplest scenario is one in which a few function calls are made within a method, each returning an error value or void.

HRESULT CFoo::DoSomething(void)

{

    HRESULT hr = E_FAIL;

 

    TraceTag((tagFoo, "CFoo::DoSomething"));

 

    CHECK_HR(DoFirstCall());

 

    CHECK_HR(DoSecondCall());

 

Cleanup:

 

    RRETURN(hr);

}

This example gives the basic structure for error handling. A trace tag is generated on entrance to the method. The tag is developer-specified; i.e., it can be turned off in most cases in order to improve debug version performance. The trace identifies the class::method being called; all methods should generate a similar trace. Notice also the use of the CHECK_HR macro wrapping calls to functions that return an HRESULT value.  The HRESULT value is returned in the Cleanup section and potentially traced by the RRETURN macro.

Remember to error check each function/method call individually. Better error handling structure can be maintained by checking the return value for each call at a time, rather than making multiple calls as part of a conditional statement.

Variations

A simple variation on this example is when an error mapping should be done. This is dependent on the type of functions/methods being called and the interface contract being fulfilled by the method itself. Generally, error mapping should be done locally unless all errors map to a single value. In the latter case, the mapping can take place in the Cleanup: block.

Scenario 2: Simple Errors with Resource Cleanup

Often, if an error occurs during a function or method, the routine will need to free some resources before returning an error code. This cleanup work should be shared among all the error conditions for the routine.

There are two basic types of routines that can fail: those that return error codes directly, and those that return a pointer indicating failure by returning a NULL pointer. The example shows how to handle this, converting NULL pointer errors to the memory error code.

HRESULT

            RGBQUAD argb[2];

        } u;

    bmi.bmih.biHeight        = _yHeight;

    bmi.bmih.biPlanes        = 1;

    bmi.bmih.biBitCount      = 1;

    bmi.u.argb[1] = g_rgbWhite;

        goto OutOfMemory;

                              hdcMem,

                              DIB_RGB_COLORS,

                              &_pvMaskBits,

                              NULL,

                              0);

    if (hdcMem)

        ReleaseMemoryDC(hdcMem);

 

    RRETURN(hr);

    hr = E_OUTOFMEMORY;

    goto Cleanup;

}

There are a couple of things to note about this example. First, any function parameters that return values to the callee are initialized in the beginning of the function if they may not return a value due to a failure. In this example, *ppBar is initialized NULL. Second, the function does not exit from the Error label; instead, the code jumps back to the Cleanup label to return, even if the Cleanup section just contains a return statement.

    TraceTag((tagFoo, "CFoo::DoMoreStuff"));

 

    *ppBar = NULL;

 

    EnterCriticalSection(&,_crit);           // Assume _crit initialized

 

    IFC(DoSomething());

 

    pBar = new CBar;

    if (pBar == NULL)

    {

        THR(E_OUTOFMEMORY);

        goto Error;

    }

 

    pData = new CData;

    if (pData == NULL)

    {

        MIL_THR(E_OUTOFMEMORY);

        goto Error;

    }

 

    pBar->pData = pData;

 

    THR(pBar->InitInstance());

    if (FAILED(hr))

    {

        TraceTag((tagError, "CFoo::DoMoreStuff - InitInstance (%lx)", hr));

        goto Error;

    }

 

    *ppBar = pBar;

 

Cleanup:

    LeaveCriticalSection(&_crit);

    RRETURN(hr);

 

Error:

    delete pBar;    // pData freed in destructor

    goto Cleanup;

}

Notice in this example that a general clean up - leaving the critical section - has to be done whether or not the method succeeds. On the other hand, the CBar pointer will be deleted only on failure. All errors proceed through the error trace tag and the critical section is always cleaned up in this example.  Note that an even better way to implement this is to create a class for the critical section that ensures that each Enter is matched with a corresponding Leave.

Also note that the only place in any of our code where it is acceptable to implement a backward jump is for the implementation of a shared cleanup section. The sort of error-handling/cleanup code illustrated here should become so common as to be idiomatic. It does not, however, condone the random use of goto statements.

Shell standard

don't use goto's or early returns to handle errors, instead nest the success conditions. this keeps the code from having to pre-init all possible variables that would need to be cleaned up in a single error collection point and generally leads to fewer lines of code in a function, resulting in more readable and maintainable code.

There are several ways of setting error values.  Two of the most common are

hr = E_FAIL;

if (condition)

{

   hr = S_OK;

}

and

if (condition)

{

   hr = S_OK;

}

else

{

   hr = E_FAIL;

}

Each of these paradigms has advantages.

The first method:

         Leads to shorter code when the same error code should be returned for errors at many levels of nesting.

         Leads to shorter code in a function with only one conditional statement.

         Allows you to put HRESULT hr and hr = E_FAIL on the same line.

         Helps keep simple code clean.

The second method:

         Reduces the likelihood of returning the wrong error code because you didnt notice an assignment further up in the nesting.

         Allows you to quickly see exactly which error code will be returned because of a failed conditional.

         Allows you to use GetLastError, which cant be done ahead of time.

         Helps keep complex code readable.

Each of these methods is appropriate in certain conditions. Developers should look at the context of the error handling code and make a decision based on that.

Parameter Validation

Do not do parameter validation in private member functions in your class implementations. Since you are the only client of those methods call them with valid parameters instead.

Do implement parameter validation in new public APIs. Be especially careful about failing calls that pass flags that are not defined or combinations of parameters that are not allowed. Not doing this limits the ability to extend the API in the future.

Do not add parameter validation to an interface implementation when there are other implementations of that interface that do not do the same level of validation. For example QueryInterface() should not validate the ppv parameter.

Do not be fooled into thinking that parameter validation is easy and trivial to add to your code. The NULL pointer is a trivial case to test for but many more forms of bad parameters exist and in the end it is impossible to test for all of them. For example a valid write pointer that points into the middle of a freed heap block is impossible to test for. Even worse is memory that changes over time (operated on by another thread).

Do set all OUT parameters when returning a success code (including S_FALSE) from your function.

When implementing an interface, do use the IFACEMETHODIMP macro. It adds the __override keyword which tells the compiler you intend to implement an interface method; the compiler will stop if it cant find the method in a base class, such as when someone changes the interface without updating your code.

Use type and buffer length safety macros

ARRAYSIZE() computes the array bounds in a type safe way that verifies that the variable is really an array. This avoids the creation of symbols or variables that track the size of an array and facilitates code that is resilient to variable array dimensions changing.

IID_PPV_ARGS() type safety macro that avoids the dangers of (void **) casts. Use with the REFIID/void ** parameter pattern common in COM programming.

IPersistFile *ppf;

hr = punk->QueryInterface(IID_PPV_ARGS(&ppf));

if (SUCCEEDED(hr))

{

    ppf->Release();

}

Releasing COM reference and Zero init

Macros like ATOMICRELEASE () will conditionally release a COM reference if it is not NULL and zero out the pointer when it is done. This is useful for cases where an interface pointer can be in either of these states. But if your code will not (or should not) produce the NULL case you should not use ATOMICRELEASE (). That is you should assert directly in your code that fact that NULL is not a valid state.

    // this is prefered

 

    IEnumUICommand *pEnumView;

    hr = LegacyEnumSpecialTasksByType(VIEWCONTROLTASKS, &pEnumView);

    if (SUCCEEDED(hr))

    {

        hr = pAggregator->AddEnum(pEnumView);

        pEnumView->Release();

    }

 

    // over this

 

    IEnumUICommand *pEnumView;

    hr = LegacyEnumSpecialTasksByType(VIEWCONTROLTASKS, &pEnumView);

    if (SUCCEEDED(hr))

    {

        hr = pAggregator->AddEnum(pEnumView);

    }

    ATOMICRELEASE(pEnumView);

The same is true for zero initing variables that will be initialized explicitly by their use an COMs convention of zeroing out params.

    IDataObject *pdo = NULL;    // zero init not needed

    hr = punk->QueryInterface(IID_PPV_ARGS(&pdo));

    if (SUCCEEDED(hr))

    {

 

        pdo->Release();

    }

 

 

    IDataObject *pdo = NULL;    // zero init not needed, failed QI will do this for you

    punk->QueryInterface(IID_PPV_ARGS(&pdo));

 

    // some code that works with our without pdo

 

    ATOMICRELEASE(pdo); // conditional release needed, pdo might be NULL

Release objects at the end of a code block, not in the middle

After you dont need an interface ref you can release it, but it is better to hold the ref to the end of the code block to match the entry, making it easy to inspect the code to make sure the release matches the initail ref.

 

    IDataObject *pdo;

    hr = punk->QueryInterface(IID_PPV_ARGS(&pdo));

    if (SUCCEEDED(hr))

    {

        pdo->DoStuff(); // last ref to pdo

 

        // more code here, does not access pdo

 

        pdo->Release(); // last code in this block, release pdo here to match the entry to this block

 

        // don't put more code after pdo->Release()

        // that makes it hard to keep track of pdo being released or not

    }

Avoid information leakage

Information hiding an important part of creating maintainable software; the less the rest of the world knows about your code the more likely you will be able to innovate in it. Given this you should follow the principal of making everything private and only exposing things publicly when absolutely necessary.

 

To achieve this you should only put a minimum amount of stuff in header files (.h) or avoid the creation of a header file at all. Only put stuff in a header when it needs to be accessed by multiple source files.

 

Specifically, when possible C++ classes should be in the .cpp file, not in a header file

 

One problem with this principal is ATL objects require their constructors to be public and visible, thus they need to be in a header. But COM objects that are supported using the class factory patterns that exists in many DLLs only require the CXXX_CreateInstance() method to be known publicly everything else can be private. See shell\inc\cfdefs.h for such a framework.

 

Of course framework classes that intended to be shared (derived from, used by) need to go into headers. But this choice should be explicitly made by the class author; most classes should not be shared. Consider using the new C++ __sealed keyword to protect classes from being derived from.

 

 

BOOL (win32) vs. bool (ansi C++)

 

bool true/false are preferred over BOOL TRUE/FALSE.

[bjervis] bool is a one byte object and BOOL is a DWORD, so I think we should start using bool (and true), since it will tend to shrink the stack frames.  We tend to use flag bits in data structures, so that stuff wont change too much.  I havent seen that the code is any worse with bool, so I dont think there is a down side.

Gotos

For Inetcore development only: gotos are only to be used with structured error handling.

For shell development only: code should be structured so gotos are not needed.

Early Returns, avoid them

Early returns are acceptable in some situations, but in general they should be avoided. Ideally functions will have a single return point at the bottom that all execution leads to. In inetcore, this should be achieved using gotos for standard error handling; in shell, the code should be carefully structured to achieve this.

Casting

Never use a C-style cast always use explicit casts

1.      Explicit casts make you think about what you are doing, provide better documentation, and are less prone to error.

o       Always be on the look out for reinterpret casts.

2.      Use DYNCAST when possible to get extra debug build validation.

o       In a free build, or if building without RTTI, DYNCAST evaluates to static_cast. In a check build, it evaluates to a function that uses RTTI information to decide whether a static_cast would succeed. If so, it does the static_cast; if not, it asserts.

3.      Explicit casts can be searched for much more easily than C-style casts.

o       Especially the dangerous reinterpret casts.

4.      Can help catch 64-bit pointer casting errors.  Example:

void *pv;

DWORD dw = 0x32b

pv = (void *)dw;                  // perfectly fine for ia64

pv = static_cast<void *>(dw);     // generates error C2440


//

// compiles okay, but did you really want to

// reinterpret 32bits as 64bits?

//

 

pv = reinterpret_cast<void *>(dw);

Reference counting

Rules for function parameters

1.      On an IN argument, the callee will not release or addref for you. (They may grab their own ref, but then they are responsible for releasing it themselves)

2.      On an OUT argument, the callee will return an addrefd reference. The caller is responsible for releasing it.

3.      On an IN/OUT argument, the callee can release the reference, replace it with another instance addref that and return it. The upshot is that the caller still owns releasing that at the end of the day.

One exception is allowed for (2): a "getter" function that doesn't addref. To avoid confusion & bugs, name such functions "GetXXXNoAddref".

Virtual functions

The destructor must be virtual for base classes

It invites bugs when a non-virtual destructor is inherited.  If a class has virtual functions, then it is most likely a base and should have a virtual destructor. (See Item 14 of "Effective C++", by Scott Meyers).

The only time it is okay to have a non-virtual destructor in a base class is when you are sure the destructor method will never be called from a context that only knows about the base class.  You had better be very sure (including the class doesn't delete itself) and it is recommend you make the destructor private.

Always use the "virtual" keyword when overriding a virtual function

Whenever you override a virtual function, use the "virtual" keyword even though C++ does not require it. (Reasons: 1. Makes it obvious when we've forgotten to make a destructor virtual. 2. Avoids a bug when you try to override a const virtual function, but forget the const. Instead of a compiler error, this causes a non-virtual overload.)

Assembly language

We occasionally have cause to use assembly language (though this is not a decision to take lightly). When we do this, maintainability and readability are particularly important, and style helps a great deal.

 

Use // comments, not ;

Theres a tug-of-war because by default, Visual Studio syntax-highlights // comments but not ; comments, while Visual SlickEdit syntax-highlights ; comments but not // comments. We resolve this in favor of // because Visual SlickEdit can be configured this way (while it appears that Visual Studio cannot.) To do this:

        Tools | Options | Color Coding

        In Lexer name select asm.

        Click the Comments tab.

        Click New Line comment.

        In Use delimiter type //

        Click OK

ATL usage

For development under shell (vs. inetcore), it is acceptable to use ATL, especially their smart pointers (these are the preferred smart pointer classes). But be aware of these problems with ATL.

1)      COM classes are constructed with a ref count of zero; in this state the objects are unstable.

2)      Class definitions need to be visible to the class factory objects, requiring them to be in header files making it hard hide this information.

3)      Destructors need to be public, making it easy to have code that deletes a COM object rather than forcing all deallocations through Release().

4)      Assumes exception based programming model. Many ATL features can not be used in non exception based code.

 

So be sure the wins you are getting out weigh these cons.

ATL and smart pointers

Smart pointers are small classes that release the reference to the object they point to in their destructor. They can be used wherever regular pointers are used, with the following caveats:

         Dont reinvent the wheel. Different smart pointer classes have slightly different semantics and using more than one leads to confusion. We use ATLs CComPtr<> and our CSmartCoTaskMemFree<>.

         Dont mix and match pointers and smart pointers in the same function. Regular pointers need to be freed or released; smart pointers dont. Using both leads to bugs where regular pointers arent released.

Editor settings

Tab Settings

In your editor, enable "expand tabs to spaces" and disable "strip trailing spaces". Otherwise, you will cause other developers to have spurious merge conflicts when you check in, and you will experience conflicts when you merge other developers' changes.

Visual SlickEdit

These features are at Tools | Options | File Options | Save.

(Prior to v9: Tools | Configuration | File Options | Save).

Source Insight:

        Enable expand tabs: Tab expansion can be enabled in Options | Document Options for each document type.

        Disable strip trailing spaces: This is disabled by default. (If you have it enabled, you can disable it at Options | Preferences | Files.)

Rascal (and presumably Visual Studio)

        Enable expand tabs: Go to Tools | Options. Click Text Editor. Click either All Languages or C/C++. Then, click Tabs, and enable Insert spaces.

        Disable strip trailing spaces: This is the default. Rascal doesnt seem to have a strip trailing spaces option.
